# Baby React Explained

이 문서는 이 프로젝트를 처음 보는 사람에게 보여주면서 설명하기 위한 README다.  
코드 전체를 다 읽지 않아도, “무엇을 만들었고 왜 이렇게 만들었는지”를 따라갈 수 있도록 정리했다.

## 이 프로젝트는 무엇인가

이 프로젝트는 **Virtual DOM 엔진 위에 React-like runtime을 직접 올린 데모 앱**이다.

결과물은 두 화면으로 구성된다.

- 왼쪽: `Baby React Emoji Reaction Board`
- 오른쪽: 런타임 디버그 패널

핵심은 단순히 이모지 앱을 만든 것이 아니라,

- 함수형 컴포넌트가 어떻게 렌더되는지
- 상태가 어떻게 유지되는지
- 상태 변경이 어떻게 diff와 patch로 이어지는지
- 그 과정이 어떻게 디버그 정보로 노출되는지

를 직접 구현하고 보여준다는 점이다.

## 화면에서 바로 볼 수 있는 것

데모 화면에서 사용자는 이모지를 눌러 반응할 수 있다.

그러면 왼쪽 앱에서는 다음이 바뀐다.

- 총 투표 수
- 현재 1위 반응
- 선두 점유율
- 최근 반응 기록

오른쪽 디버그 패널에서는 다음이 바뀐다.

- 마지막 액션 로그
- 어떤 컴포넌트가 다시 렌더됐는지
- 어떤 patch가 만들어졌는지
- 마지막 런타임 노드 로그

즉, 사용자 입장에서는 “클릭했더니 숫자가 바뀌었다”로 끝나지만, 이 프로젝트는 그 뒤의 내부 흐름까지 같이 보여준다.

## 이 프로젝트가 보여주려는 핵심 개념

### 1. 상태는 루트에만 둔다

이번 프로젝트는 `root-only Hook` 정책을 따른다.

- 루트 컴포넌트가 상태를 가진다
- 자식 컴포넌트는 props만 받는다
- 자식 컴포넌트에서 Hook은 쓰지 않는다

이렇게 하면 데이터 흐름이 단순해지고, 상태 변경이 어떤 렌더를 만들었는지 설명하기 쉬워진다.

### 2. 함수형 컴포넌트는 바로 DOM으로 가지 않는다

함수형 컴포넌트는 먼저 `COMPONENT_NODE`라는 중간 표현으로 만들어진다.  
그다음 `resolveVNodeTree()`가 이 노드를 실제 렌더 가능한 트리로 푼다.

그 이후에야 아래 단계가 진행된다.

- `diff(oldTree, newTree)`
- `applyPatches()`
- 실제 DOM 반영

즉, 이 프로젝트의 핵심 렌더 흐름은 다음과 같다.

```text
User Action
-> setState
-> Root Re-render
-> COMPONENT_NODE 생성
-> resolveVNodeTree()
-> diff()
-> applyPatches()
-> DOM 업데이트
```

### 3. 디버그 패널은 런타임 내부를 직접 읽지 않는다

디버그 패널은 런타임 객체의 내부 배열이나 필드를 직접 읽지 않는다.  
오직 아래 공개 API만 사용한다.

- `getDebugSnapshot()`
- `subscribeDebug()`

이렇게 한 이유는 디버그 UI와 runtime 구현을 느슨하게 연결하기 위해서다.  
즉, 패널은 “내부 구현”이 아니라 “관찰 가능한 공개 상태”를 구독하는 구조다.

## 이 프로젝트의 동작을 한 번에 이해하는 방법

### 1. 사용자가 이모지를 누른다

이모지 버튼 클릭은 루트 상태를 바꾼다.

- `votes` 증가
- `selectedEmoji` 변경
- `recentReactions` 갱신
- `lastAction` 갱신

### 2. 루트가 다시 렌더된다

루트 컴포넌트는 현재 상태를 바탕으로 새 트리를 만든다.  
이 과정에서 파생 지표도 계산된다.

예를 들면 이런 값들이다.

- `totalVotes`
- `topReaction`
- `topPercentage`
- `visibleRecentReactions`

### 3. 이전 트리와 비교한다

새 트리는 이전 트리와 `diff()`된다.  
이때 텍스트가 바뀌었는지, props가 바뀌었는지, 노드가 추가/삭제됐는지가 patch로 정리된다.

### 4. 최소한의 DOM 변경만 적용한다

`applyPatches()`는 전체를 다시 그리지 않고 바뀐 부분만 DOM에 반영한다.

즉, 이 프로젝트는 “상태가 바뀌면 무조건 화면 전체를 갈아끼운다”가 아니라  
“상태가 바뀌면 새 트리를 계산하고, 필요한 DOM만 최소 수정한다”를 보여준다.

### 5. 같은 변화가 디버그 패널에도 나타난다

런타임은 렌더 횟수, 마지막 액션, 렌더 추적, 패치 요약을 snapshot으로 정리한다.  
디버그 패널은 이 snapshot을 구독해서 오른쪽 레일에 보여준다.

그래서 발표할 때 아래처럼 연결해서 설명할 수 있다.

- 사용자가 무엇을 했는가: 액션 로그
- 어떤 컴포넌트가 다시 계산됐는가: 렌더 추적
- 실제 DOM 변경이 무엇이었는가: 패치 요약

## 브라우저 저장소는 왜 들어가 있나

이 프로젝트는 현재 라이브 상태와 저장된 상태를 구분해서 보여준다.

- `Save`: 현재 상태를 `localStorage`에 저장
- `Reset`: 현재 라이브 상태만 초기화
- `Restore`: 저장본을 다시 현재 화면에 적용

이 기능 덕분에 단순 렌더링 데모를 넘어서,

- 상태 직렬화
- 저장과 복구
- 화면 상태와 영속 상태의 차이

까지 같이 설명할 수 있다.

## 왜 이 프로젝트가 의미가 있나

보통 React를 배우면 컴포넌트를 “사용”하는 쪽에 익숙해진다.  
하지만 이 프로젝트는 반대로,

- 상태가 어디에 저장되는지
- Hook이 왜 호출 순서에 의존하는지
- 왜 새 트리를 만든 뒤 비교하는지
- 왜 diff 결과를 patch로 바꾸는지

를 구현하는 쪽에서 이해하게 만든다.

즉, “React를 쓴 앱”이 아니라 **React의 핵심 아이디어를 축소해서 직접 만든 시스템**이라는 점이 중요하다.

## 남에게 설명할 때 추천하는 순서

1. 왼쪽 앱과 오른쪽 디버그 패널 구조를 먼저 보여준다.
2. 이모지를 눌러서 앱 상태와 디버그 정보가 동시에 바뀌는 것을 보여준다.
3. 상태는 루트에만 있고 자식은 props만 받는다고 설명한다.
4. `COMPONENT_NODE -> resolveVNodeTree() -> diff -> patch` 흐름을 말해준다.
5. 디버그 패널은 `getDebugSnapshot()` / `subscribeDebug()`만 쓴다고 강조한다.
6. 마지막에 저장/복구를 보여주며 브라우저 저장소 개념까지 연결한다.

## 짧게 요약하면

이 프로젝트는 이모지 보드를 만든 것이 목적이 아니라,

- 상태 관리
- 렌더 트리 생성
- diff/patch
- 디버그 관찰

을 한 화면 안에서 설명 가능한 형태로 묶은 작은 React-like runtime 실험이다.
