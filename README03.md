# Baby React, Explained Through the App

이 문서는 이 프로젝트를 처음 보는 사람에게 보여주면서 설명하기 위한 README다.  
체크리스트를 나열하기보다, 실제로 동작하는 앱을 따라가면서 이 프로젝트가 어떤 구조로 만들어졌는지 이해할 수 있도록 구성했다.

## 이 프로젝트를 한 문장으로 말하면

이 프로젝트는 **기존 Virtual DOM 엔진 위에, 함수형 컴포넌트와 Hook 런타임을 직접 올려서 만든 React-like 웹 앱**이다.

결과물은 두 화면으로 구성된다.

- 왼쪽: `Baby React Emoji Reaction Board`
- 오른쪽: 런타임 디버그 패널

왼쪽은 사용자가 실제로 상호작용하는 앱이고, 오른쪽은 그 상호작용이 런타임 내부에서 어떤 렌더와 patch를 만들었는지 보여주는 관찰 화면이다.

즉, 이 프로젝트는 “예쁜 데모 앱” 하나를 만드는 데서 끝나지 않고,  
**Component, State, Hooks, Virtual DOM, Diff/Patch**가 한 흐름으로 이어지는 모습을 브라우저에서 직접 보여주는 데 집중했다.

## 화면을 보면 무엇이 보이나

사용자는 이모지를 눌러 반응할 수 있다.  
그 순간 왼쪽 앱에서는 다음 정보가 함께 바뀐다.

- 총 투표 수
- 현재 1위 반응
- 선두 점유율
- 최근 반응 기록
- 저장 시각과 현재 액션 상태

오른쪽 디버그 패널에서는 같은 변화가 다른 관점으로 보인다.

- 마지막 액션 로그
- 렌더 추적
- 패치 요약
- 마지막 런타임 노드 로그

이 구조 덕분에 “사용자가 클릭했다”는 사실이,

- 앱 상태를 바꾸고
- 다시 렌더를 만들고
- 최소 DOM patch를 만들고
- 그 결과가 디버그 정보로 드러나는 과정

까지 자연스럽게 연결된다.

## 컴포넌트는 어떻게 나뉘어 있나

이 프로젝트의 컴포넌트는 작은 화면 단위로 나누어져 있다.  
예를 들면 네비게이션, 메트릭 카드, 이모지 보드, 최근 활동, 저장 컨트롤, 디버그 패널이 각각 함수형 컴포넌트로 분리되어 있다.

다만 중요한 점은, 이 컴포넌트들이 각자 상태를 따로 들고 있지 않다는 점이다.

- 루트 컴포넌트가 상태를 관리한다
- 자식 컴포넌트는 props만 받아서 화면을 그린다
- 자식은 가능한 한 순수 함수에 가깝게 유지한다

그래서 화면은 여러 조각으로 나뉘어 있지만, 데이터 흐름은 한 방향으로 유지된다.  
이 덕분에 “어디서 값이 바뀌는가”가 명확하고, 설명도 쉬워진다.

## 상태는 어디에 있고 왜 그렇게 두었나

상태는 루트 컴포넌트인 `EmojiReactionBoardApp`에 집중되어 있다.

대표적인 루트 상태는 다음과 같다.

- `votes`
- `selectedEmoji`
- `recentReactions`
- `savedAt`
- `lastAction`
- 디버그 레일 관련 UI 상태

이 값들은 사용자의 입력이나 클릭으로 실제로 변하는 원본 데이터다.

반면 아래 값들은 상태를 그대로 저장한 것이 아니라, 루트 상태로부터 계산되는 파생 지표다.

- `totalVotes`
- `topReaction`
- `topPercentage`
- `visibleRecentReactions`

즉, 이 프로젝트는 “원본 상태는 루트에 두고, 필요한 화면 값은 계산해서 사용한다”는 구조를 택하고 있다.  
이런 방식은 상태를 여러 곳에 중복 저장하지 않아도 되고, 다시 렌더될 때 어떤 값이 왜 바뀌는지 설명하기 편하다.

## Hooks는 어떤 문제를 풀고 있나

이 프로젝트에서 Hooks의 핵심 질문은 이것이다.

> 함수는 매번 다시 실행되는데, 상태는 어떻게 유지될까?

여기서는 그 문제를 `FunctionComponent` 인스턴스가 해결한다.  
이 인스턴스는 루트 컴포넌트를 감싸고 있고, 내부에 `hooks[]` 배열과 `hookIndex`를 두고 있다.

즉, 컴포넌트 함수가 다시 실행되더라도 상태는 함수 바깥의 인스턴스에 남아 있고,  
Hook은 호출 순서를 기준으로 자기 자리를 다시 찾아간다.

이 런타임에서 실제로 사용하는 Hook은 세 가지다.

- `useState`
- `useEffect`
- `useMemo`

여기에 디버그 기록용 보조 훅도 붙어 있지만, 기본적인 상태/사이드이펙트/메모이제이션의 축은 위 세 가지다.

## setState는 값만 바꾸는 게 아니다

이 프로젝트에서 `setState`는 단순히 값을 교체하는 함수가 아니다.  
값이 바뀌면 그 다음에 무엇을 해야 하는지도 같이 책임진다.

흐름은 대략 이렇다.

```text
setState
-> hook slot 값 갱신
-> update 예약
-> 루트 다시 렌더
-> 새 Virtual DOM 생성
-> diff
-> patch 적용
```

즉, 상태 변경은 곧 화면 갱신의 시작점이다.

그리고 같은 이벤트 안에서 여러 상태가 연속으로 바뀌는 경우를 대비해,  
업데이트는 microtask 기반으로 한 번에 묶는다.  
덕분에 여러 `setState`가 연달아 호출돼도 렌더가 불필요하게 여러 번 반복되지 않는다.

## 함수형 컴포넌트는 어떻게 렌더되나

이 프로젝트는 함수형 컴포넌트를 만나자마자 DOM으로 보내지 않는다.

먼저 함수형 컴포넌트는 `COMPONENT_NODE`라는 중간 표현으로 만들어진다.  
그다음 `resolveVNodeTree()`가 이 노드를 실제 렌더 가능한 트리로 풀어준다.

그 이후부터는 기존 Virtual DOM 엔진이 맡는다.

- 새 트리 생성
- 이전 트리와 비교
- 바뀐 부분만 patch로 정리
- 실제 DOM에 최소 반영

이 과정을 두 단계로 나눈 덕분에,

- 컴포넌트 경계는 유지하면서
- 실제 렌더 단계는 단순한 트리 비교로 처리할 수 있다

는 장점이 생긴다.

## 왜 Diff/Patch가 중요한가

상태가 바뀔 때마다 화면 전체를 통째로 다시 그리는 것도 가능하다.  
하지만 이 프로젝트는 일부만 바뀌는 경우가 많기 때문에, 가능한 한 필요한 부분만 바꾸는 방식을 유지한다.

그래서 런타임은 새 Virtual DOM을 만든 뒤, 이전 트리와 비교해서 patch를 계산한다.  
예를 들면 이런 변화들이다.

- 텍스트만 바뀜
- props만 바뀜
- 노드가 추가됨
- 노드가 제거됨
- 노드가 교체됨

이런 구조 덕분에 사용자는 자연스러운 화면 갱신을 보고,  
우리는 그 뒤에서 어떤 최소 변경이 일어났는지 설명할 수 있다.

## 디버그 패널은 왜 넣었나

이 프로젝트의 차별점은 오른쪽 디버그 패널이다.

보통 데모 앱은 “결과 화면”만 보여주지만, 여기서는 런타임 내부 변화도 별도 패널로 같이 보여준다.  
특히 다음 세 가지는 발표에서 설명력이 좋다.

- 액션 로그: 방금 어떤 사용자 입력이 들어왔는가
- 렌더 추적: 어떤 컴포넌트가 다시 렌더되었는가
- 패치 요약: 어떤 patch가 만들어졌는가

중요한 건 이 패널이 내부 자료구조를 직접 뒤지는 게 아니라,  
`getDebugSnapshot()`과 `subscribeDebug()`라는 공개 API만 사용한다는 점이다.

즉, 디버그 패널은 runtime에 붙어 있는 또 하나의 화면이지만,  
설계 자체는 “관찰 가능한 상태를 구독하는 독립 클라이언트”에 가깝다.

## 브라우저 저장소는 왜 들어가 있나

이모지 보드는 단순 클릭 데모를 넘어서, 저장과 복구 흐름도 갖고 있다.

- `Save`: 현재 보드 상태를 브라우저 저장소에 저장
- `Reset`: 현재 라이브 상태만 초기화
- `Restore`: 마지막 저장 상태를 다시 현재 화면에 적용

이 흐름을 통해 이 프로젝트는 단순 렌더링뿐 아니라,

- 상태 직렬화
- 저장된 스냅샷 복구
- 현재 메모리 상태와 영속 상태의 차이

까지 함께 보여준다.

## 구현 재료는 의도적으로 단순하게 유지했다

이 프로젝트는 외부 프레임워크 없이 브라우저 환경에서 직접 동작하도록 만들었다.

- JavaScript
- HTML
- CSS

만 사용하고, React나 Vue 같은 프레임워크는 쓰지 않았다.

그 이유는 “비슷한 결과를 만드는 것”보다  
“핵심 동작을 직접 구현해서 설명할 수 있는가”가 더 중요했기 때문이다.

## 실제 React와 비교하면

이 프로젝트는 React와 동일한 구현은 아니다.  
대신 React의 핵심 아이디어를 이해하기 좋은 크기로 줄여서 보여준다.

예를 들면 다음과 같은 차이가 있다.

- Hook은 루트에서만 사용한다
- 상태도 루트에서만 관리한다
- 더 복잡한 scheduler 대신 단순한 batching 구조를 쓴다
- 디버그 정보가 발표용으로 잘 드러나도록 구성돼 있다

즉, 완전히 같은 제품을 복제한 것이 아니라,  
핵심 원리를 관찰 가능하게 만든 학습용 구현이라고 보는 편이 맞다.

## 검증은 어떻게 했나

이 프로젝트는 화면만 동작하게 만든 것이 아니라, 테스트로도 검증했다.

- 런타임 테스트
- diff 관련 테스트
- 디버그 패널 테스트
- 데모 상호작용 테스트

즉, 컴포넌트 렌더, Hook 동작, patch 생성, 사용자 시나리오까지 단계별로 확인할 수 있게 해 두었다.

## 이 프로젝트를 남에게 설명할 때 좋은 순서

1. 왼쪽 앱과 오른쪽 디버그 패널 구조를 먼저 보여준다.
2. 이모지를 눌러 상태 변화와 화면 변화를 보여준다.
3. 상태는 루트에만 있고 자식은 props만 받는다고 설명한다.
4. `FunctionComponent`, `hooks[]`, `mount()`, `update()` 개념을 연결한다.
5. `COMPONENT_NODE -> resolve -> diff -> patch` 흐름을 짚는다.
6. 디버그 패널이 액션/렌더/패치를 어떻게 보여주는지 설명한다.
7. 마지막에 `Save / Reset / Restore`로 상태 저장과 복구를 보여준다.

## 짧게 요약하면

이 프로젝트는 단순한 이모지 앱이 아니라,

- 함수형 컴포넌트
- 루트 상태 관리
- Hook 런타임
- Virtual DOM 비교
- 최소 DOM patch
- 디버그 관찰 도구

를 하나의 브라우저 경험으로 묶어낸 작은 React-like 시스템이다.

```bash
npm ci
npm run dev
```

검증 명령:

```bash

```
