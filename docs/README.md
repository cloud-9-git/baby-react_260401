# 5-Minute Presentation README

이 문서는 발표 화면에 띄워두고 그대로 따라가는 5분용 진행표다.  
발표는 `README + 데모앱 시연` 중심으로 진행하고, 코드 파일은 필요할 때만 짧게 언급한다.

## 1. 우리가 만든 것

`Baby React Emoji Reaction Board`는 우리가 직접 만든 React-like runtime 위에서 동작하는 데모 앱이다.

이번 발표의 핵심은 세 가지다.

- 함수형 컴포넌트와 Hook 런타임을 직접 구현했다.
- `COMPONENT_NODE -> resolveVNodeTree() -> diff/applyPatches` 흐름으로 화면이 갱신된다.
- 디버그 패널이 `getDebugSnapshot()` / `subscribeDebug()`만으로 런타임 변화를 보여준다.

## 2. 5분 발표 순서

### 0:00 - 0:40 소개

- 기존 Virtual DOM 엔진 위에 React-like runtime을 올린 프로젝트라고 소개한다.
- 오늘은 “상태 변경이 렌더와 patch로 어떻게 이어지는지”를 보여준다고 말한다.

### 0:40 - 2:20 데모 시연

- 이모지를 2번 정도 눌러 총 투표 수, 1위 반응, 최근 반응 변화를 보여준다.
- 오른쪽 디버그 패널 `액션 로그`에서 `react` payload가 바뀌는 것을 보여준다.
- `렌더 추적` 탭에서 어떤 컴포넌트가 다시 렌더됐는지 보여준다.
- `패치 요약` 탭에서 실제 patch가 기록되는 것을 보여준다.
- `Save -> Reset -> Restore`를 빠르게 보여준다.

### 2:20 - 3:40 구현 포인트 설명

- 상태는 루트 컴포넌트에만 두고, 자식은 props만 받는다고 설명한다.
- 함수형 컴포넌트는 바로 DOM으로 보내지 않고 `COMPONENT_NODE`로 만든 뒤 `resolveVNodeTree()`로 푼다고 설명한다.
- 이후 resolved tree 기준으로 `diff`와 `applyPatches`가 동작한다고 정리한다.

### 3:40 - 4:20 디버그 패널 설명

- 패널은 런타임 내부 배열을 직접 읽지 않는다고 강조한다.
- `getDebugSnapshot()` / `subscribeDebug()`만으로 액션, 렌더 추적, 패치 요약을 구독한다고 설명한다.
- 즉, UI와 runtime을 느슨하게 연결한 공개 API 설계라고 정리한다.

### 4:20 - 5:00 테스트와 마무리

- 테스트와 빌드 검증까지 끝났다고 말한다.
- 마지막에 “이 프로젝트는 React 스타일 UI가 아니라 runtime 자체를 구현하고 설명 가능한 형태로 만든 결과물”이라고 정리한다.

## 3. 시연 중 꼭 보여줄 것

- 왼쪽은 사용자 데모 앱, 오른쪽은 디버그 패널이라는 구조
- 이모지 클릭 후 총 투표 수 / 1위 반응 / 최근 반응 동시 갱신
- 디버그 패널의 `react` 액션 payload
- `렌더 추적`과 `패치 요약`
- `Save`, `Reset`, `Restore`

## 4. 발표에서 꼭 말할 문장

### 상태 구조

“이번 라운드는 root-only Hook 정책이라서 상태는 루트에만 두고, 자식 컴포넌트는 props만 받아 렌더합니다.”

### 렌더 흐름

“함수형 컴포넌트는 바로 DOM으로 가는 게 아니라 `COMPONENT_NODE`를 거치고, `resolveVNodeTree()` 이후에야 diff와 patch가 적용됩니다.”

### 디버그 패널

“디버그 패널은 런타임 내부를 직접 읽지 않고 공개 debug snapshot API만 사용합니다.”

### 마무리

“저희는 React를 사용하는 데서 멈추지 않고, 상태 변경이 patch와 debug 정보로 이어지는 과정을 직접 구현해봤습니다.”

## 5. 발표 중 예상 질문

### 왜 Hook을 자식 컴포넌트에서 막았나요?

이번 목표는 Hook 분산보다 runtime 구조를 명확하게 설명하는 것이 더 중요했기 때문이다.  
루트 하나에 상태를 집중하면 상태 변화, 렌더, patch 흐름을 훨씬 선명하게 보여줄 수 있다.

### 왜 디버그 패널을 별도로 만들었나요?

데모 UI와 관찰 UI를 분리해도 공개 API만 있으면 연결할 수 있다는 점을 보여주기 위해서다.

### 왜 MOVE patch는 안 넣었나요?

이번 라운드는 key와 component node를 안정적으로 도입하는 것이 우선이었고, reorder 최적화까지 동시에 넣으면 복잡도가 커지기 때문이다.

## 7. 데모가 꼬였을 때

- `Reset`으로 현재 라이브 상태를 비운다.
- 저장본이 있으면 `Restore`로 복구한다.
- 시간이 부족하면 디버그 패널과 테스트 결과를 보여주며 내부 동작 설명으로 전환한다.
