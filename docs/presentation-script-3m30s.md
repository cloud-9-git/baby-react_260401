# 3분 30초 발표 스크립트

## 0:00 - 0:20 시작

안녕하세요. 저희는 이번 과제에서 React의 핵심 개념인 Component, State, Hooks를 직접 구현했습니다.  
그리고 그 구현이 실제로 동작한다는 것을 보여주기 위해, 이모지 반응 보드 데모까지 만들었습니다.

## 0:20 - 0:45 우리가 어떻게 만들었는가

저희 구현의 핵심은 아주 단순합니다.  
루트 컴포넌트 하나가 상태와 Hook을 모두 관리하고, 자식 컴포넌트들은 상태 없이 props만 받아서 화면만 그립니다.  
즉, 상태는 한곳에 모으고, 화면은 역할별로 나눠서 구성했습니다.

## 0:45 - 1:10 왜 이렇게 했는가

이렇게 만든 이유는 과제 조건에 맞추기 위해서입니다.  
이번 과제는 Hook을 루트에서만 사용하고, 자식은 stateless하게 만들라는 조건이 있었습니다.  
그래서 저희는 Lifting State Up 구조를 가장 분명하게 보여주는 방식으로 설계했습니다.

## 1:10 - 1:40 상태는 어떻게 유지되는가

중요한 질문은 함수형 컴포넌트가 매번 다시 실행되는데 상태가 어떻게 남아 있느냐입니다.  
저희는 상태를 함수 안이 아니라 `FunctionComponent` 안의 `hooks[]` 배열에 저장했습니다.  
그리고 렌더할 때마다 Hook을 같은 순서로 다시 읽게 해서, 함수는 다시 실행돼도 상태는 유지되도록 만들었습니다.

## 1:40 - 2:05 setState와 batching

또 `setState`는 값을 바꾸는 것에서 끝나지 않습니다.  
값이 정말 바뀌었는지 확인하고, 실제로 바뀌었을 때만 다음 렌더를 예약합니다.  
그리고 같은 순간에 여러 상태가 바뀌어도 `queueMicrotask()`를 이용해 렌더는 한 번만 일어나게 했습니다.  
이게 저희가 구현한 batching입니다.

## 2:05 - 2:35 화면은 어떻게 바뀌는가

상태가 바뀌면 루트 컴포넌트가 다시 실행되고, 새 Virtual DOM이 만들어집니다.  
그다음 이전 Virtual DOM과 새 Virtual DOM을 비교해서 바뀐 부분만 찾고, 그 부분만 실제 DOM에 반영합니다.  
즉, 전체 화면을 다시 그리는 게 아니라 필요한 부분만 고칩니다.

## 2:35 - 3:10 데모

이제 데모를 보겠습니다.  
이모지를 하나 클릭하면 총 투표 수, 현재 1위 반응, 최근 활동이 같이 바뀝니다.  
이건 루트 상태 하나가 바뀌고, 그 결과를 여러 자식 컴포넌트가 다시 받아서 화면에 보여주기 때문입니다.  
오른쪽 디버그 패널을 보면 액션 로그, 렌더 추적, patch 요약도 함께 바뀝니다.  
즉, 내부에서 어떤 일이 일어났는지를 눈으로 확인할 수 있습니다.  
그리고 `Save`, `Reset`, `Restore` 버튼으로 상태 저장, 초기화, 복원 흐름도 보여줄 수 있습니다.

## 3:10 - 3:25 실제 React와의 차이

물론 이 구현은 실제 React와 완전히 같지는 않습니다.  
실제 React는 모든 함수형 컴포넌트가 state를 가질 수 있고, 더 복잡한 스케줄러와 Fiber 구조를 사용합니다.  
반면 저희는 그중 핵심인 상태 보존, Hook 순서, batching, Diff/Patch만 설명하기 쉬운 구조로 압축했습니다.

## 3:25 - 3:30 마무리

정리하면, 저희 프로젝트는 React를 그대로 복제한 것이 아니라, React가 왜 그렇게 동작하는지를 직접 구현으로 보여주는 결과물입니다.
