# 3분 30초 발표 스크립트

## 0:00 - 0:25 소개

안녕하세요. 5조 발표 시작하겠습니다. 
이번 과제에서 React의 핵심 기능인 Component, State, Hooks를 직접 구현하고, 그 위에 실제로 동작하는 웹 페이지를 만들었습니다.

## 0:25 - 0:55 전체 구조

전체 구조는 간단합니다.  
`FunctionComponent` 클래스가 루트 함수형 컴포넌트를 실행하고, `hooks[]` 배열에 상태를 저장합니다.  
상태가 바뀌면 새 Virtual DOM을 만들고, 이전 Virtual DOM과 `diff`한 다음, `applyPatches`로 실제 DOM의 바뀐 부분만 갱신합니다.

## 0:55 - 1:25 요구사항 대응

이번 과제의 제약도 그대로 반영했습니다.  
Hook은 루트 컴포넌트에서만 사용 가능하게 만들었고, 자식 컴포넌트는 상태 없이 `props`만 받는 stateless 함수로 구현했습니다.  
즉, 상태는 루트에 두고 자식은 화면 표현만 담당하는 **Lifting State Up 구조**입니다.

## 1:25 - 1:55 Hook 설명

핵심 질문은 함수형 컴포넌트가 매번 다시 실행되는데 상태가 어떻게 유지되느냐입니다.  
저희는 `FunctionComponent` 인스턴스 안에 `hooks[]` 배열을 두고, Hook 호출 순서대로 상태를 저장했습니다.  
그래서 다음 렌더에서도 같은 순서의 Hook은 같은 슬롯을 다시 읽고, 상태가 유지됩니다.

## 1:55 - 2:20 setState와 batching

또 `setState`는 값만 바꾸는 함수가 아닙니다.  
값을 바꾼 뒤 바로 렌더하지 않고 update를 예약하고, 같은 tick 안의 여러 상태 변경을 `queueMicrotask()`로 한 번에 묶습니다.  
즉, 상태가 여러 번 바뀌어도 렌더는 한 번만 일어나게 해서 batching을 구현했습니다.

## 2:20 - 3:05 데모

이제 데모를 보겠습니다.  
이모지를 하나 클릭하면 총 투표 수, 현재 1위 반응, 최근 활동이 동시에 바뀝니다.  
이 변화는 모두 루트 상태가 바뀌면서 자식 컴포넌트들이 props를 다시 받아 렌더된 결과입니다.  
오른쪽 디버그 패널을 보면 마지막 액션, 렌더 추적, patch 요약도 함께 갱신됩니다.  
즉, 사용자 입력에서 상태 변경, Virtual DOM diff, 실제 DOM patch까지의 흐름을 한 화면에서 확인할 수 있습니다.  
이어서 `save`를 누르면 현재 상태가 브라우저 저장소에 저장되고, `reset`을 누르면 라이브 상태만 초기화됩니다.  
마지막으로 `restore`를 누르면 저장했던 상태가 다시 복원됩니다.

## 3:05 - 3:25 실제 React와의 차이

물론 저희 구현은 실제 React와 완전히 같지는 않습니다.  
Fiber, concurrent rendering, child component Hook 같은 기능은 없습니다.  
하지만 그 대신 React의 본질인 **상태 유지, rerender, diff, patch, batching**을 훨씬 직접적으로 보여줍니다.

## 3:25 - 3:30 마무리

정리하면, 저희 프로젝트는 React를 사용하는 것이 아니라 React의 핵심 원리를 직접 구현하고 설명할 수 있도록 만든 결과물입니다.
