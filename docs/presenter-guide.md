# Presenter Guide

## 이 파일의 목적

이 문서는 발표자가 아래 두 가지를 동시에 할 수 있게 돕는 자료입니다.

1. 쉬운 말로 설명하기
2. 그 쉬운 말이 코드에서 정확히 무엇을 뜻하는지 이해하기

즉, 발표할 때는 쉽게 말하고, 질문을 받으면 이 문서를 보고 정확하게 답할 수 있도록 만든 자료입니다.

---

## 1. 가장 먼저 말할 핵심

### 짧게 말하면

> "우리는 기존 Virtual DOM 엔진 위에, 루트 컴포넌트 하나가 상태와 Hook을 관리하는 React-like runtime을 만들었습니다."

### 이 말의 정확한 뜻

- `FunctionComponent`가 루트 컴포넌트를 감쌉니다.
- 상태는 `FunctionComponent.hooks[]` 배열에 저장됩니다.
- `useState`, `useEffect`, `useMemo`는 이 배열을 이용해 동작합니다.
- 상태가 바뀌면 새 Virtual DOM을 만들고 `diff -> applyPatches`로 화면을 갱신합니다.

### 발표에서 왜 중요한가

이 한 문장에 이 프로젝트의 차별점이 거의 다 들어 있습니다.  
다른 팀도 같은 과제를 했기 때문에, "우리는 어떤 구조를 선택했는가"를 먼저 보여주는 것이 중요합니다.

---

## 2. 우리 팀의 핵심 선택

| 질문 | 우리 팀의 선택 | 쉽게 설명하면 |
| --- | --- | --- |
| 상태는 어디에 두었나? | 루트 컴포넌트 | 상태를 한곳에 모아둠 |
| 자식 컴포넌트는 어떤가? | props-only 순수 함수 | 화면만 그리는 부품 |
| Hook 값은 어디에 남나? | `hooks[]` 배열 | 메모장 칸처럼 저장 |
| 상태가 바뀌면? | update 예약 | 바로 그리지 않고 다음 차례에 그림 |
| 여러 상태가 한 번에 바뀌면? | batching | 한 번만 다시 그림 |
| 화면은 어떻게 바뀌나? | Diff 후 Patch | 바뀐 부분만 고침 |

---

## 3. 러버덕에게 설명하듯 말하는 버전

### 컴포넌트 구조

쉬운 설명:

> "큰 상자 하나가 데이터를 다 들고 있고, 작은 상자들은 받은 데이터를 화면에 보여주기만 합니다."

정확한 설명:

- 루트 컴포넌트가 상태와 이벤트를 모두 관리합니다.
- 자식 컴포넌트는 state 없이 `props`만 받아 렌더링합니다.
- 이 구조는 과제의 Lifting State Up 조건을 그대로 반영합니다.

왜 이렇게 했는가:

- 과제에서 Hook은 루트에서만 사용 가능했기 때문입니다.
- 상태 흐름이 단방향이어서 설명이 쉽습니다.
- 구현 복잡도가 크게 낮아집니다.

---

### Hook 상태 유지

쉬운 설명:

> "함수는 다시 실행되지만, 상태는 함수 밖 메모장에 적어둡니다."

정확한 설명:

- 상태는 함수 안이 아니라 `FunctionComponent.hooks[]` 배열에 저장됩니다.
- 렌더할 때마다 `hookIndex`를 0부터 다시 시작합니다.
- 같은 순서의 Hook은 같은 배열 칸을 다시 사용합니다.

왜 이렇게 했는가:

- Hook의 본질인 "호출 순서 기반 상태 보존"을 가장 간단하게 보여줄 수 있기 때문입니다.

주의할 점:

- Hook 순서가 바뀌면 다른 칸을 읽게 되므로 구조가 깨집니다.

---

### `setState`

쉬운 설명:

> "`setState`는 값을 바꾸는 버튼이 아니라, 다음 그림을 예약하는 버튼입니다."

정확한 설명:

- 새 값과 이전 값을 비교합니다.
- 값이 달라졌을 때만 Hook 슬롯을 갱신합니다.
- 그 다음 `scheduleUpdate()`를 호출해 다음 렌더를 예약합니다.

왜 이렇게 했는가:

- 상태 변경과 UI 갱신을 자동으로 연결하기 위해서입니다.
- 같은 값을 다시 넣었을 때 불필요한 렌더를 막기 위해서입니다.

---

### batching

쉬운 설명:

> "한 번에 바뀐 것들은 모아서 한 번만 다시 그립니다."

정확한 설명:

- `queueMicrotask()`와 `isUpdateScheduled` 플래그를 사용합니다.
- 같은 tick 안에서 여러 `setState`가 호출돼도 `update()`는 1번만 실행됩니다.

왜 이렇게 했는가:

- 렌더 횟수를 줄이기 위해서입니다.
- React의 batching 개념을 가장 짧게 구현할 수 있기 때문입니다.

대안:

- 즉시 update
- `setTimeout`
- `requestAnimationFrame`
- 더 큰 스케줄러 구조

---

### Diff/Patch

쉬운 설명:

> "화면 전체를 새로 그리는 대신, 바뀐 부분만 찾아서 고칩니다."

정확한 설명:

- 상태가 바뀌면 새 Virtual DOM을 만듭니다.
- 이전 Virtual DOM과 새 Virtual DOM을 비교합니다.
- 바뀐 부분만 patch 목록으로 만들고 실제 DOM에 반영합니다.

왜 이렇게 했는가:

- 과제 목표가 전체 재렌더가 아니라 필요한 부분만 업데이트하는 것이기 때문입니다.
- 기존에 만든 Virtual DOM 엔진을 그대로 활용할 수 있기 때문입니다.

## 4. 발표할 때 꼭 강조해야 할 포인트

### 포인트 1. 우리는 상태를 루트에 모았다

이 말이 중요한 이유:

- 과제 조건을 가장 직접적으로 만족합니다.
- 다른 팀과 비교할 때 구조적 선택이 바로 드러납니다.

### 포인트 2. 우리는 Hook을 `hooks[]` 배열로 구현했다

이 말이 중요한 이유:

- "함수는 다시 실행되는데 상태는 어떻게 유지되나?"라는 질문에 바로 답이 됩니다.

### 포인트 3. 우리는 batching을 넣었다

이 말이 중요한 이유:

- 단순 구현을 넘어서 React의 중요한 동작 개념까지 반영했다는 뜻입니다.

### 포인트 4. 우리는 Debug Panel로 내부 동작을 보여준다

이 말이 중요한 이유:

- 다른 팀도 데모는 만들 수 있지만, 내부 동작까지 시각화한 점은 차별점이 됩니다.

말할 문장:

> "지금 클릭 한 번으로 총 투표 수, 현재 1위, 최근 활동이 같이 바뀝니다. 이건 루트 상태 하나가 바뀌고, 자식 컴포넌트들이 그 결과를 다시 받기 때문입니다."

말할 문장:

> "오른쪽 패널을 보면 액션, 렌더 추적, patch 요약이 함께 갱신됩니다. 즉, 화면만 바뀌는 것이 아니라 내부 동작도 같이 확인할 수 있습니다."

### Save / Reset / Restore

말할 문장:

> "이 버튼들로 상태 저장, 초기화, 복원을 보여주면서 state 흐름이 실제로 동작한다는 것을 확인할 수 있습니다."

## 6. 예상 질문과 짧은 답변

### Q. 왜 자식 컴포넌트에서 Hook을 못 쓰게 했나요?

A. 과제 제약을 그대로 반영했고, 상태 흐름을 단순하게 유지하기 위해서입니다.

### Q. 왜 상태를 다 루트에 몰았나요?

A. Lifting State Up을 가장 명확하게 보여줄 수 있고, 구현이 단순해지기 때문입니다.

### Q. 왜 `hooks[]` 배열을 썼나요?

A. Hook 순서 기반 상태 보존이라는 핵심 원리를 가장 작은 구조로 설명할 수 있기 때문입니다.

### Q. 실제 React와 가장 큰 차이는 무엇인가요?

A. 실제 React는 모든 함수형 컴포넌트가 state를 가질 수 있고, Fiber와 더 복잡한 스케줄러를 사용합니다.

### Q. 왜 Debug Panel을 만들었나요?

A. 내부에서 실제로 렌더와 patch가 일어난다는 것을 시각적으로 보여주기 위해서입니다.
