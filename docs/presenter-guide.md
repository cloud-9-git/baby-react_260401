# Presenter Guide

이 문서는 `README.md`와 발표 스크립트의 순서를 그대로 따라가면서,  
발표자가 각 문장을 **쉽게 말하되 정확하게 이해할 수 있도록** 돕는 설명 자료입니다.

구성 순서도 `README`와 같습니다.

1. 무엇을 구현했는가
2. 어떻게 만들었는가

## 1. 무엇을 구현했는가

### 발표에서 이렇게 말하면 된다

> "React의 핵심 개념인 Component, State, Hooks를 직접 구현하고, 그 구현이 실제로 동작한다는 것을 보여주는 데모까지 만들었습니다."

### 정확한 뜻

- 단순히 UI만 만든 것이 아니라 런타임 자체를 직접 구현했습니다.
- 핵심은 `FunctionComponent`, `useState`, `useEffect`, `useMemo`, `diff`, `applyPatches`입니다.
- 데모 앱은 이 구현이 실제로 동작한다는 증거 역할을 합니다.

### 이 문장이 중요한 이유

첫 문장에서 "React를 사용한 앱"이 아니라 "React-like runtime을 직접 구현한 결과물"임을 분명히 해야 합니다.

## 2. 어떻게 만들었는가

### 발표에서 이렇게 말하면 된다

> "루트 컴포넌트 하나가 상태와 Hook을 모두 관리하고, 자식 컴포넌트들은 state 없이 props만 받아 화면만 그리도록 만들었습니다."

### 정확한 뜻

- 루트 컴포넌트가 상태와 이벤트를 모두 가집니다.
- 자식 컴포넌트는 stateless한 순수 함수입니다.
- 화면은 역할별로 나누고, 데이터는 위에서 아래로만 흐릅니다.

### 왜 이렇게 했는가

- 과제 조건이 Hook을 루트에서만 사용하도록 제한했기 때문입니다.
- 구현 복잡도를 낮추면서 Lifting State Up을 분명하게 보여줄 수 있습니다.

### 질문이 들어오면

Q. 왜 자식 state를 안 만들었나요?  
A. 과제 제약을 그대로 반영했고, 데이터 흐름을 단순하게 유지하기 위해서입니다.

---

## 3. 왜 이렇게 만들었는가

### 발표에서 이렇게 말하면 된다

> "이번 과제의 핵심 제약은 Hook은 루트에서만 쓰고, 자식은 stateless하게 유지하는 것이었습니다. 그래서 상태를 위로 올리는 구조를 가장 명확하게 보여주는 방식으로 설계했습니다."

### `README`와 연결되는 선택 포인트

| 선택 포인트 | 설명할 핵심 |
| --- | --- |
| 루트 + props-only 자식 | 상태와 화면 책임을 분리 |
| 상태 루트 집중 | Hook 제약을 가장 분명하게 만족 |
| Debug Panel | 내부 동작도 같이 보여주는 차별점 |

### 발표 포인트

- 다른 팀과 비교될 때 가장 먼저 보이는 것은 "구조를 어떻게 잡았는가"입니다.
- 이 프로젝트의 차별점은 기능 개수보다 **구조 선택이 분명하다**는 점입니다.

---

## 4. 상태는 어떻게 유지되는가

### 발표에서 이렇게 말하면 된다

> "함수형 컴포넌트는 렌더마다 다시 실행되지만, 상태는 함수 안이 아니라 `FunctionComponent` 안의 `hooks[]` 배열에 저장했습니다."

### 정확한 뜻

- 상태는 함수 바깥의 배열에 저장됩니다.
- 렌더가 다시 일어나면 `hookIndex`를 0부터 다시 올립니다.
- 같은 순서의 Hook이 같은 배열 칸을 다시 사용합니다.

### 왜 이렇게 했는가

- Hook의 본질이 "호출 순서 기반 상태 보존"이기 때문입니다.
- 배열과 인덱스만으로도 이 개념을 가장 단순하게 보여줄 수 있습니다.

### 질문이 들어오면

Q. 왜 `hooks[]` 배열을 썼나요?  
A. Hook 순서 기반 상태 보존을 가장 작은 구조로 설명할 수 있기 때문입니다.

Q. Hook 순서가 바뀌면 어떻게 되나요?  
A. 다른 슬롯을 읽게 되므로 구조가 깨지고, 현재 구현에서는 에러로 간주합니다.

## 5. `setState`와 batching은 어떻게 동작하는가

### 발표에서 이렇게 말하면 된다

> "`setState`는 값을 바꾸는 것에서 끝나지 않고, 값이 실제로 바뀌었을 때만 다음 렌더를 예약합니다. 그리고 같은 순간의 여러 변경은 한 번만 렌더되도록 batching했습니다."

### 정확한 뜻

- `setState`는 값 비교를 합니다.
- 값이 달라졌을 때만 Hook 슬롯을 갱신합니다.
- 그 다음 `scheduleUpdate()`로 다음 렌더를 예약합니다.
- `queueMicrotask()`와 `isUpdateScheduled`로 여러 변경을 1번으로 묶습니다.

### 왜 이렇게 했는가

- 불필요한 렌더를 막기 위해서입니다.
- React의 batching 개념을 과제 수준에서 간단히 재현할 수 있기 때문입니다.

### 질문이 들어오면

Q. 왜 바로 update하지 않았나요?  
A. 같은 이벤트 안에서 여러 상태가 바뀌면 렌더가 너무 많이 일어나기 때문입니다.

Q. 왜 `queueMicrotask()`를 썼나요?  
A. 구현이 단순하고, 같은 tick의 변경을 한 번으로 묶기 적절하기 때문입니다.

---

## 6. 화면은 어떻게 바뀌는가

### 발표에서 이렇게 말하면 된다

> "상태가 바뀌면 루트 컴포넌트가 다시 실행되고, 새 Virtual DOM을 만든 뒤 이전 Virtual DOM과 비교해서 바뀐 부분만 실제 DOM에 반영합니다."

### 정확한 뜻

- 새 Virtual DOM 생성
- 이전 Virtual DOM과 `diff`
- patch 생성
- `applyPatches()`로 실제 DOM 최소 수정

### 왜 이렇게 했는가

- 과제 목표가 "전체를 다시 그리지 않고 필요한 부분만 업데이트"였기 때문입니다.
- 기존 Virtual DOM 엔진을 그대로 재사용할 수 있었습니다.

### 질문이 들어오면

Q. 왜 전체 DOM을 다시 만들지 않았나요?  
A. 과제의 핵심이 필요한 부분만 업데이트하는 구조를 직접 구현하는 데 있기 때문입니다.

Q. key 기반 비교도 있나요?  
A. 현재 구현은 단순 비교 중심이고, 일부 keyed diff만 지원합니다.

## 7. 데모에서 보여주는 것

### 발표에서 이렇게 말하면 된다

> "이 데모는 예쁜 화면을 보여주는 것이 아니라, 루트 상태 변경과 렌더 흐름이 실제로 동작한다는 것을 눈으로 확인하는 장치입니다."

### 데모별 설명 문장

#### 이모지 클릭

> "클릭 한 번으로 총 투표 수, 현재 1위, 최근 활동이 같이 바뀝니다. 하나의 루트 상태 변경이 여러 UI로 퍼지는 구조를 보여줍니다."

#### Debug Panel

> "오른쪽 패널에서는 액션 로그, 렌더 추적, patch 요약까지 같이 보입니다. 즉, 화면 변화뿐 아니라 내부 동작도 함께 확인할 수 있습니다."

#### Save / Reset / Restore

> "저장, 초기화, 복원을 통해 state 흐름과 side effect가 실제로 동작한다는 것을 확인할 수 있습니다."

### 질문이 들어오면

Q. 왜 Debug Panel을 넣었나요?  
A. 런타임이 실제로 어떻게 동작하는지 시각적으로 보여주기 위해서입니다.

## 8. 실제 React와의 차이

### 발표에서 이렇게 말하면 된다

> "현재 구현은 실제 React를 완전히 복제하는 것이 아니라, 상태 보존, Hook 순서, batching, Diff/Patch 같은 핵심 개념을 설명하기 쉬운 구조로 압축한 버전입니다."

### 꼭 기억할 차이

| 항목 | 현재 구현 | 실제 React |
| --- | --- | --- |
| 상태 단위 | 루트 FunctionComponent 하나 | 각 함수 컴포넌트마다 독립 상태 |
| Hook 사용 위치 | 루트만 허용 | 모든 함수형 컴포넌트 |
| 상태 저장 | `hooks[]` 배열 | Fiber 기반 Hook 구조 |
| batching | `queueMicrotask` 기반 | 더 정교한 scheduler |
| 렌더 비교 | vDOM diff 후 patch | Fiber reconciliation |

### 질문이 들어오면

Q. 실제 React와 가장 큰 차이는 무엇인가요?  
A. 실제 React는 모든 함수형 컴포넌트가 state를 가질 수 있고, Fiber와 더 정교한 스케줄러를 사용합니다.
