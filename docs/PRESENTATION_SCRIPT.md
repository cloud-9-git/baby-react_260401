# 5-Minute Presentation Script

이 문서는 `docs/README.md`를 화면에 띄워 놓고 읽는 흐름에 맞춘 5분 대본이다.  
발표 중에는 README의 섹션 제목을 따라가고, 실제 브라우저 데모를 옆에서 같이 보여준다.

## 0:00 - 0:40 오프닝

안녕하세요. 저희는 이번 프로젝트에서 React의 핵심 동작을 직접 구현해보는 것을 목표로 했습니다.  
최종 결과물은 `Baby React Emoji Reaction Board`이고, 왼쪽은 사용자 데모 앱, 오른쪽은 같은 런타임을 관찰하는 디버그 패널입니다.

오늘은 이 화면 하나로 세 가지를 보여드리겠습니다.  
첫째, 함수형 컴포넌트와 Hook 런타임을 직접 만들었다는 점.  
둘째, 상태 변경이 `COMPONENT_NODE`, resolve, diff, patch 흐름으로 이어진다는 점.  
셋째, 디버그 패널이 공개 snapshot API만으로 런타임 변화를 보여준다는 점입니다.

## 0:40 - 2:20 데모 시연

먼저 실제 상호작용부터 보겠습니다.  
이모지를 누르면 루트 상태의 투표 수가 바뀌고, 총 투표 수와 1위 반응, 최근 반응이 함께 갱신됩니다.

여기서 중요한 점은 상태가 여러 군데 흩어진 것처럼 보여도 실제 상태는 루트 컴포넌트가 관리한다는 것입니다.  
자식 컴포넌트는 상태를 가지지 않고 props만 받아서 렌더합니다.

이제 오른쪽 디버그 패널을 보겠습니다.  
`액션 로그` 탭에서는 방금 어떤 사용자 액션이 발생했는지와 payload를 볼 수 있습니다.  
지금처럼 `react` 액션과 눌린 이모지 정보가 기록됩니다.

`렌더 추적` 탭에서는 이 상호작용 때문에 어떤 컴포넌트들이 다시 렌더됐는지 확인할 수 있습니다.  
즉, 클릭이 바로 DOM 변경으로 가는 게 아니라 먼저 상태를 바꾸고, 그 상태를 기준으로 컴포넌트 트리가 다시 계산된다는 흐름을 볼 수 있습니다.

`패치 요약` 탭에서는 마지막 diff 결과가 어떤 patch로 번역됐는지 확인할 수 있습니다.  
텍스트가 바뀌었는지, props가 바뀌었는지, 경로가 어디인지를 바로 보여줍니다.

마지막으로 `Save`, `Reset`, `Restore`를 보겠습니다.  
`Save`는 현재 상태를 저장하고, `Reset`은 현재 라이브 상태만 초기화하며, `Restore`는 마지막 저장본을 다시 현재 화면으로 복구합니다.

## 2:20 - 3:40 구현 포인트

이제 구현 포인트를 짧게 설명드리겠습니다.

첫째, Hook 정책은 `root-only`입니다.  
그래서 상태는 루트에만 두고, 자식 컴포넌트는 props만 받아 렌더합니다.  
이 구조 덕분에 상태 변화와 렌더 흐름을 훨씬 명확하게 설명할 수 있습니다.

둘째, 함수형 컴포넌트는 바로 DOM으로 가지 않습니다.  
먼저 `createElement()`가 함수 타입을 만나면 `COMPONENT_NODE`를 만들고,  
`resolveVNodeTree()`가 그것을 실제 `ELEMENT_NODE`와 `TEXT_NODE` 트리로 풀어줍니다.

그 다음에야 resolved tree 기준으로 `diff`와 `applyPatches`가 동작합니다.  
즉, 이번 프로젝트의 핵심 흐름은 `상태 변경 -> render -> resolve -> diff -> patch`입니다.

## 3:40 - 4:20 디버그 패널 포인트

디버그 패널도 중요한 포인트입니다.  
이 패널은 런타임 내부 배열이나 필드를 직접 읽지 않습니다.

대신 `getDebugSnapshot()`과 `subscribeDebug()`라는 공개 API만 사용해서  
마지막 액션, 렌더 추적, 패치 요약을 받아와서 그립니다.

즉, 디버그 UI와 runtime 내부 구현을 강결합하지 않고, 관찰 가능한 공개 인터페이스를 별도로 설계했다는 점이 이번 구조의 장점입니다.

## 4:20 - 5:00 테스트와 마무리

마지막으로, 저희는 화면만 만든 것이 아니라 테스트와 빌드 검증도 마쳤습니다.  
2026년 4월 2일 기준으로 `npm test -- --run`은 14개 테스트 파일, 77개 테스트가 모두 통과했고, `npm run build`도 성공했습니다.

정리하면, 이번 프로젝트는 React처럼 보이는 UI를 하나 만든 것이 아니라,  
상태 변화가 컴포넌트 트리와 patch, 그리고 debug 정보로 이어지는 과정을 직접 구현하고 설명 가능한 형태로 만든 작은 React-like 시스템입니다.

감사합니다.

## 짧은 예비 엔딩 멘트

“저희는 React를 사용하는 것이 아니라, React의 핵심 아이디어를 직접 구현하는 쪽에서 이해해보자는 목표로 이번 프로젝트를 진행했습니다.”

## 시연이 꼬였을 때 이어가는 멘트

“라이브 상태는 `Reset`과 `Restore`로 바로 복구할 수 있고, 디버그 패널과 테스트 결과를 통해 내부 동작은 계속 설명드릴 수 있습니다.”
